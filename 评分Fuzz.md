## 摘要

本文提出一种**基于指令类别评分并结合测试用例评估**的模糊测试方案。首先对待测系统的指令进行语义分类，并基于漏洞触发概率、影响范围、执行成本、可控性、语义多样性等维度为每类指令赋予初始评分。然后结合生成算法（如随机生成、变异、语义引导或语法模板生成）产生测试用例，并按照指令评分加权决定生成策略概率。测试用例执行后，根据覆盖率、异常触发等反馈计算用例得分，并应用阈值策略决定执行顺序。动态过程中，统计异常指令及其置信度，通过衰减或学习率调整类别评分，平衡探索与利用。最后通过设计对照实验评估覆盖率、缺陷发现率、执行效率等指标。

本方案目标是**设计一个可实施的模糊测试框架**，通过对指令类别的评分来指导测试用例生成和选择，同时结合动态反馈优化策略，尽可能提高漏洞触发率与测试效率。



### 指令分类方法与初始评分策略

#### 指令分类

首先根据待测目标的业务场景或静态分析，将所有可能的指令调用按功能或风险分为若干类别。例如，对于通用程序，可以按**功能类别**（算术运算、内存访问、分支控制、文件/网络I/O等）或**权限级别**（用户态调用、系统调用）等进行划分。通用分类：算术指令、逻辑指令、内存读写、控制流、系统调用等类别。

#### 初始评分维度

对每个指令类别赋予多个评分维度，综合计算初始风险权重。包括：

- **触发漏洞概率**：该类别指令历史上触发漏洞的可能性（高危操作优先）。可参考类似CVSS漏洞评分框架进行定性赋分。
- **影响范围**：指令对程序状态的影响程度（如能修改全局状态或安全关键资源则影响范围大）。
- **执行成本**：执行该指令/用例的时间或资源消耗（高成本操作可适当降低优先级，以提高效率）。
- **可控性**：模糊测试能多大程度控制该指令的输入参数（可控性高意味着更易于构造相关用例）。
- **语义多样性**：该指令类别在语义上的丰富性或独特性（多样性高可引入更多测试场景）。
- **异常历史**：历史测试中该类别导致过异常的次数或严重程度。

将上述维度归一化后组合（可采用加权平均）得到每个类别的初始综合评分。



### 测试用例生成

测试用例生成策略包括：

- **突变生成：** 以一组有效种子为基础，通过字节/字段级别的随机变异产生新输入。
- **语法/模板生成：** 利用目标系统输入格式的语法或模板生成合法输入，保证较高的有效率。

生成时使用 **加权随机 + 探索机制**：

```
P(inst) = (1 - ε) * exploitation + ε * exploration
```

- exploitation：按评分选
- exploration：随机选

ε 可以逐渐降低（类似强化学习）



本方案采用**混合策略**：每轮根据**指令类别评分**动态调整生成概率。具体地，在生成测试用例时将视作多轮决策过程：不断从指令类别池中采样下一个指令，生成一个指令序列作为测试用例。采样时令高评分的类别具有更高被选中的概率，**偏好高风险类别**但仍保留低风险类别一定概率（保持多样性，避免陷入局部）。



### 测试用例评分规则

每个指令 `inst` 有一个评分：

```
Score(inst) = α * BaseScore
            + β * BugScore
            + γ * CoverageScore
            + δ * NoveltyScore
```

**各部分含义：**

- **BaseScore**：人工经验（初始值）
- **BugScore**：历史触发异常频率
- **CoverageScore**：带来的路径覆盖提升
- **NoveltyScore**：是否探索新行为（避免重复）

每个测试用例的**整体评分**可由其包含的指令类别得分和执行结果综合计算：

- 令用例包含的指令序列为
  $$
  {I_1,I_2,\dots}
  $$
  ，初始可令用例基础分为各指令类别得分之和（或加权平均）：
  $$
  S_{\text{base}}=\sum_i S_{\text{instr}(I_i)}
  $$
  
- 执行反馈影响：若用例在执行过程中**触发异常**（崩溃、挂起、断言失败），可加上一个较高的异常分值；若未触发异常但**覆盖了新路径**，可加入覆盖增量分值；否则，可能仅保留基础分或适当衰减。

- 可选添加熵或多样性因素：若输入多样性高，也可以略微加分，鼓励探索不同的输入空间。

  《MulCovFuzz》

- 覆盖率：多覆盖率加权融合

- 覆盖效率：单位时间覆盖收益最大化

例如，可以定义最终得分公式：
$$
S_{\text{case}} = \beta,S_{\text{base}} + \gamma,(\text{newCoverage}) + \delta,(\text{crashFlag})
$$
其中
$$
\beta,\gamma,\delta为权重参数，
$$
`newCoverage`为新增覆盖率指标，`crashFlag`为0/1指示是否崩溃。也可采用更复杂的函数或学习模型来综合评分。



### 阈值策略

| 分数区间 | 策略           |
| -------- | -------------- |
| 高分     | 优先执行       |
| 中分     | 正常执行       |
| 低分     | 降低概率但保留 |

可以用一个**优先队列（priority queue）**：

```
priority = TestCaseScore
```

执行策略：

- 高分优先
- 但定期执行低分（避免局部最优）

将测试用例得分与预设阈值比较：若低于阈值，则视为低价值用例，可立即丢弃或放入低优先级队列，避免浪费资源。高于阈值的用例则加入测试队列，后续可能再被用于变异或重点测试。阈值可动态调整（如保留分值排名前K%的用例），以兼顾稳定性与效率。也可以采用**优先级队列**，始终优先执行得分高的用例，确保资源集中于潜在高风险路径。



### 动态评分更新与异常反馈机制

在执行过程中不断收集反馈，并根据异常情况调整指令类别评分：

- **异常指令统计：** 若某测试用例触发崩溃或异常，根据执行轨迹可以定位导致异常的指令。对该类别的指令增大权重或提升评分。

- **覆盖置信度：** 若用例覆盖了新代码块，对涉及的指令类别也提高信心分；长期未发现新覆盖的指令类别可逐渐衰减评分，避免过度利用已探索区域。

- **置信度与学习率：** 引入学习率更新评分，例如：
  $$
  S_{\text{new}} = (1-\alpha)S_{\text{old}} + \alpha \times \text{observedReward}
  $$
  其中observedReward可以根据是否触发漏洞或覆盖进行赋值。类似多臂老虎机（MAB）算法中的更新策略，可平衡长期和短期效果。

- **避免过拟合与探索/利用：** 为防止始终只选热门类别导致探索空间缩小，可引入
  $$
  \epsilon-贪婪策略或随机化成分
  $$
  ，即一定概率随机选取较低评分类别来保持多样性。同时对持续未见异常的类别进行衰减（降低得分），对长期高得分类别增加不确定度因子（减少过度偏向），保持**探索-利用平衡**。

具体地，假设类别i的当前评分为S_i，当一个含有类别i的用例触发异常（视为成功），可以更新：

```
S_i ← S_i + learning_rate * (reward - S_i)
```

其中reward可为1（崩溃）或覆盖增量，learning_rate逐渐减小防止振荡。若反复试验无结果，则可对S_i乘以衰减因子<1。这样，类别评分随时间自适应调整。

通过上述动态机制，模糊测试器能“学习”哪些指令序列更可能触发漏洞，并逐渐偏重这类输入，同时偶尔尝试其他选项。多臂老虎机理论和强化学习已被证明适用于种子调度和变异选择。例如，T-Scheduler将种子调度建模为MAB问题，通过汤普森采样自动平衡探索与利用，不依赖手工调参。
